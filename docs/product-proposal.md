# Product proposal (PoC)

Целевое видение и сценарии; **текущая реализация** описана в [README.md](../README.md) и [system-design.md](system-design.md).

## Идея

Планирование поездки объединяет цены на перелёты, отели и содержательную часть маршрута. PoC показывает агентный пайплайн: извлечение параметров → API → генерация Markdown-плана → проверка бюджета (LangGraph).

## Цель PoC

- принять запрос на естественном языке;
- получить структурированные параметры (`TripQuery`);
- вызвать Travelpayouts для рейсов/отелей и LLM для достопримечательностей и итогового маршрута;
- вернуть пользователю читаемый план и оценку бюджета.

## Метрики успеха (ориентиры)

- пользователь получает связный план по дням;
- при заданном бюджете срабатывает guardrail или показывается предупреждение при исчерпании ретраев;
- инструменты вызываются осмысленно (см. тесты и ручные прогоны).

Точные SLO (latency p95, доля успешных сценариев) в коде не зафиксированы.

## Основной сценарий

Пользователь: «Поездка в Барселону на 4 дня, бюджет 1000€».

Ожидаемый поток: извлечение дат и города → поиск рейсов/отелей (если область поиска не ограничена только отелями/рейсами) → подсказки мест → генерация маршрута → сверка с бюджетом.

## Edge cases (частично покрыты)

- **Превышение бюджета:** граф уменьшает эффективный лимит для отелей и повторяет `fetch_data` до лимита ретраев.
- **Нет рейсов / ошибка API:** ответы тулов с ошибкой и пустыми списками; пользователь видит пояснение.
- **Слишком низкий бюджет:** ранний выход на этапе `validate`.

## Что не входит в PoC

Бронирование, оплата, погодный API, отдельный RAG-индекс достопримечательностей, долговременная память пользователя.

## Роль LLM

- извлечение структуры запроса и достопримечательностей;
- генерация текстового маршрута.

Вызовы внешних travel-API выполняются кодом (Travelpayouts), не «галлюцинациями» модели. Проверка бюджета в графе — программная (`node_guardrail`).
