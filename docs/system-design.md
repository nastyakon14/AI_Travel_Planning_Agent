## Архитектура AI Travel Planning Agent (PoC)

### 1. Ключевые архитектурные решения
*   **Orchestration:** Использование **LangGraph** в качестве фреймворка. Это позволяет задать жесткий цикл выполнения (циклический граф), контролировать состояние (State) на каждом шаге и делать ретраи при падении инструментов.
*   **Agent Paradigm:** Используется паттерн **Stateful Workflow (Plan-and-Execute)**. Главный агент (Planner) разбивает запрос на подзадачи, вызывает нужные тулы, а затем агрегирует результаты, сверяясь с бюджетом.
*   **LLM Engine:** OpenAI GPT-4o-mini (для быстрых вызовов инструментов) и GPT-5.1 (для финального синтеза маршрута).
*   **Memory:** Краткосрочная память сессии реализуется через встроенный чекпоинтинг LangGraph поверх **PostgreSQL/Redis**.
*   **Retrieval (RAG):** Использование **Vector DB** (Qdrant/Chroma/FAISS) для хранения базы знаний о достопримечательностях и правилах визового режима.

### 2. Список модулей и их роли
*   **API Gateway:** Принимает запросы, управляет сессиями.
*   **Orchestrator (LangGraph):** Управляет потоком выполнения, маршрутизирует задачи между узлами графа.
*   **State Manager:** Хранит контекст поездки (даты, локации, оставшийся бюджет, найденные билеты).
*   **Tool Executor Layer:** Изолированная среда для вызова внешних API (Авиа, Отели, Погода).
*   **Guardrail/Validator Module:** Проверяет ответы LLM (например, не превышен ли бюджет программно, а не только логикой LLM).
*   **Retriever:** Ищет локальные достопримечательности по векторной базе.

### 3. Основной Workflow выполнения задачи
1.  **Extract Intent:** Пользователь пишет запрос -> LLM извлекает `origin, destination, dates, budget`.
2.  **State Update:** Обновление глобального словаря состояния агента.
3.  **Flight Search Node:** Вызов API авиабилетов -> запись вариантов в State -> вычет из виртуального бюджета.
4.  **Hotel Search Node:** Вызов API отелей -> запись в State -> обновление бюджета.
5.  **Attraction Retrieval Node:** Поиск мест по VectorDB -> добавление в контекст.
6.  **Itinerary Generation:** LLM формирует маршрут по дням, учитывая собранные данные.
7.  **Validation (Guardrail):** Проверка, что сумма билетов и отелей $\le$ начального бюджета. Если нет -> возврат к шагу 3 с промптом "найди дешевле".
8.  **Output:** Отправка пользователю markdown-ответа.

### 4. Описание state / memory / context handling
*   **Session State:** Каждый запрос привязан к `thread_id`. LangGraph сохраняет `TypedDict` состояния (намерения, история сообщений, промежуточные JSON от API).
*   **Memory Policy:** В рамках PoC хранится только история текущего планирования поездки (окно диалога). После формирования Travel Plan контекст очищается, сохраняется только сам план.
*   **Context Budgeting:** Ограничение окна контекста до 16k токенов. Длинные JSON-ответы от API фильтруются на уровне Tool Executor (LLM отдаются только топ-3 самых релевантных и дешевых билета, а не весь дамп API).

### 5. Failure Modes, Fallback и Guardrails
*   **Галлюцинации LLM (выдумала отель):** *Guardrail* — финальный маршрут должен ссылаться только на `hotel_id` и `flight_id`, полученные из State. Если найдена выдумка — retry узла генерации.
*   **API недоступно (Timeout):** *Fallback* — использование закешированных "mock" данных (в рамках PoC) или информирование пользователя: "Не удалось получить цены на перелет, ориентировочная стоимость X".
*   **Неадекватные требования (Рим за 50$):** Ранний выход (Early Exit). На этапе *Extract Intent* проверяется адекватность. Агент возвращает ответ: "Увеличьте бюджет" без вызова API.

### 6. Ограничения (SLAs)
*   **Latency:** До 1 минуты на полный цикл планирования (допустимо для асинхронного агента).
*   **Cost:** До 0.05$ за одну поездку (за счет фильтрации JSON перед подачей в LLM).
*   **Reliability:** 75% успешных парсингов сложных запросов.
